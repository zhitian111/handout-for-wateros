# 引言
从最开始学习计算机导论我们就一直在写程序，从C语言到C++，再到我们现在要使用的Rust，我们知道，这些都是高级语言。而我们还知道，计算机只能“读”的懂指令，也就是一条一条的机器码。从本质上来看，如果我们不考虑外设，计算机的在做的其实就是使用一定的时间，按照我们定义的工作方式，将内存中的数据修改成我们预期的样子。那在冯诺依曼架构下，我们所定义的工作方式（程序对应的机器码）也是存储在内存中的，我们所使用的数据也是存储在内存中的。那我们就可以说，程序其实就是特定的被我们定义出来内存空间，我们指定了这片空间的初始状态，以及哪一部分存储数据，哪一部分存储指令。
> 这里的说法并不准确，因为程序（program）严格来说指的是那个你双击之后或者是使用命令之后可以运行的文件，上方的这段描述应该说是进程（process）概念的抽象描述。Anyway,我们不在乎这些概念的准确含义，你们只需要知道我这段话在描述什么就可以了。

## 程序的存储时布局
### 可执行文件
可执行文件的本质其实就是按照一定的规则向读取它的对象（操作系统或者机器）描述了程序在运行时的内存布局和入口点。
而我们的操作系统执行一个程序，其实就是按照这个可执行文件的描述设置好了内存空间，并控制CPU从程序入口点开始执行。

只是听讲解可能有些抽象，我们直接执行一个程序来查看一下真实的情况。

# 可执行文件的构建
OK，那有了以上这些知识之后，我们就知道，我们写代码，编译程序，其实就是为了获得这个可执行文件。
你们可能还记得以前咱上的一些课里让我们执行的命令，比如要编译一个C语言的程序，你可以直接使用
```bash
gcc xxx.c -o xxx
```
然后就可以直接获得一个叫做"xxx"的可执行文件，但实际上这是因为gcc本身就打包了很多工具，所以使得整个过程过程直接自动化了。
但是从源文件到可执行文件，实际上有许多任务要做。
首先我们需要对源文件进行预处理，最常见的预处理就是宏展开，比如我们在写C语言的时候的#define的展开，在写rust的时候println!()这里的!表示的其实也是一个宏。对于我们编写者来说这就够了。但对于真正为我们完成这项工作的工具来说，这其中还会包含语法分析、中间代码生成等，中间代码回携带更多的信息，以便于编译器进行处理。
cargo为我们提供了查看以上这些内容的工具，不过对于我们来说，查看一下宏展开的结果用于调试一些宏就足够的了。
```bash
cargo rustc -- -Zunpretty=expanded
```
或者也可以使用专门的工具
```bash
cargo install cargo-expand # 安装expand工具
cargo expand # 展开整个项目中的所有宏
cargo expand module_name::function_name # 展开某个模块的某个方法中的宏
```

预处理之后，程序会由编译器进行编译，这一步是获取目标文件（二进制文件，一般为.o或者.obj后缀）或者机器码的中间表示形式（汇编代码，一般为.s后缀），而如果生成的是汇编代码则会由汇编器再生成目标文件，实际上现代的编译器大部分情况下都是直接生成的目标文件，只是原则上会有汇编这一步。
## 目标文件
这是我们对于我们这个项目来说非常重要的一种文件。这类文件中包含了一下内容：
- **机器码**：源代码翻译成的目标机器的指令。
- **数据段**：程序中定义的全局变量、静态变量等数据。
- **符号表**：记录函数、变量等符号的名称及其地址，用于后续链接。
- **重定位信息**：记录需要链接器调整的地址引用（如函数调用或变量访问）。
- **调试信息**（可选）：用于调试器的信息，如源代码行号与机器码的对应关系。
当然，借助现代的工具我们可以不在意这个文件里实际的布局，但我们必须要知道，这个文件中存储了我们的程序的实际机器码和符号地址对应表。
还记的上次我讲的吗，我们编写程序的本质其实是在定义各种符号。这个符号表的对应关系其实就是我们的变量、函数名等和其在文件中地址（偏移量）的对应关系。它描述了这个符号在这个文件中的位置，让我们可以通过符号名在这个文件中找到它。
那这么做有什么用呢？
用处很大。从工程的角度上来讲，这可以让我们重用很多模块，以此来减少代码编译产生开销，比如我们有三个源文件，其中的两个源文件都使用到了另一个源文件里的符号，我们就可以通过目标文件作为中转，对于这个共用的源文件，我们只需要编译一次就可以了。另外，因为每个源文件都可以对应一个目标文件，我们可以分别将所有源文件都编译成目标文件，在对目标文件进行排列组合，由此来获取不同的可执行文件。并且因为大部分编译型语言都会经过目标文件这一步，或者至少你可以要求编译器输出目标文件，我们可以在一个项目里混用不同的编程语言，只需要按照一定规则来存放相同类型的数据和进行调用就可以了。这也就是为什么在我们的项目里会有汇编代码生成目标文件这一步。
而完成由目标文件到可执行文件这一步的工具，叫做链接器。
关于静态库和动态库的链接，因为我们写这个内核并不需要链接其他的库，这里就不赘述了。

## 链接器
还记的我们说过的吗，可执行文件描述的了程序运行时的内存布局。它只是做一个描述的工作，只负责携带这些信息，对吧。至于你（操作系统/机器）读取到了之后要怎么把它们放到物理内存里，这是你们的事儿，不是我这个文件负责的对吧。

那我是不是就可以放飞自我了。比如说我们有一个.text段，我们不把它放在0x0这个地址，我们把它放在0x8200_0000这个地址，我只管这么描述，你怎么放我不管。而所谓的段，他其实也就是一个符号而已，那我是不是可以说我有很多很多的段，我有.A.text段，有.B.text段，只是你操作系统（.exe）要求我是这四个段而已啊，我又不在你（Windows）上边执行，我为什么要管你的规则。

没错，我们可以完完全全自己定义最终获取到的可执行文件的中所携带的信息，包括但不限于程序段数、段名，符号地址，段地址。并且，我们知道，连接器是把一些符号对应的地址根据对应表填过去对吧，但是我填写到程序里的地址，跟我实际放的地址有什么关系呢？我完全可以说aaa这个符号在0xffff_0000_0000，而我这个符号实际上放的地址是0x0，这个作用我会在讲解虚拟内存的时候详细说明。

OK，那我们该怎么指定这些事情呢？这就需要链接器脚本来完成了。
### 链接器脚本
链接器脚本（linker script）其实也是一种语言，由链接器解析，指导链接器完成可执行文件的构建工作。那我在这儿只讲一些我会的，其实我了解的也不多，但是足够让我们完成这些工作了。

#### 概述
链接器脚本的主要功能是：
- **描述内存布局**：定义目标系统的内存区域（如 ROM、RAM）及其属性。
- **控制段映射**：指定输入文件的段（section，如 `.text`、`.data`、`.bss`）如何映射到输出文件的段。
- **设置符号地址**：为符号（如变量、函数入口）分配特定的内存地址。
- **其他功能**：如定义入口点、控制填充、对齐、垃圾回收等。

链接脚本通常以 `.ld` 或 `.lds` 为扩展名，使用 `ld` 的 `-T` 选项指定，例如：
```bash
ld -T linkerscript.ld input.o -o output.elf
```
如果未指定，`ld` 使用默认脚本（可用 `ld --verbose` 查看）。
#### 基本语法规则
链接脚本的语法类似 C 语言，具有以下特点：
- **语句分隔**：语句以分号 `;` 分隔。
- **注释**：使用 `/* */` 风格注释，例如 `/* This is a comment */`。
- **表达式**：支持 C 风格的算术和逻辑运算，支持十六进制（如 `0x1000`）、十进制、移位运算等。
- **定位器符号**：`.` 表示当前地址（位置计数器），用于跟踪输出文件中的地址分配。
- **大小写敏感**：符号名、段名等区分大小写。
- **通配符**：支持 `*`（任意字符）、`?`（单个字符）、`[a-z]`（字符范围）等通配符，用于匹配文件或段名。
##### 链接脚本的主要组成部分
链接脚本由一系列命令和赋值语句组成，核心命令包括 `MEMORY`、`SECTIONS`、`ENTRY` 等。以下逐一讲解。

##### MEMORY 命令
`MEMORY` 命令定义目标系统的内存区域，指定每个区域的起始地址、长度和属性。

**语法**：
```ld
MEMORY
{
  name [(attr)] : ORIGIN = origin, LENGTH = len
  name [(attr)] : ORIGIN = origin, LENGTH = len
  ...
}
```

- **name**：内存区域的名称，自定义且唯一，仅在链接脚本中有效。
- **attr**：可选的属性字符串，指定内存区域的权限：
  - `R`：只读（Read-only）
  - `W`：读写（Read/Write）
  - `X`：可执行（Executable）
  - `A`：可分配（Allocatable）
  - `I` 或 `L`：已初始化
  - `!`：反转上述属性的含义
  - 例如：`(rx)` 表示只读且可执行，适合存放代码；`(rwx)` 表示可读写执行，适合 RAM。
- **ORIGIN**：内存区域的起始地址（可缩写为 `org` 或 `o`）。
- **LENGTH**：内存区域的长度（可缩写为 `len` 或 `l`），支持单位如 `K`（1024 字节）、`M`（1024K 字节）。

**示例**：
```ld
MEMORY
{
  FLASH (rx) : ORIGIN = 0x08000000, LENGTH = 1024K  /* Flash for code */
  RAM (rwx)  : ORIGIN = 0x20000000, LENGTH = 128K   /* RAM for data */
}
```
此例定义了两个内存区域：`FLASH`（只读可执行，1MB）和 `RAM`（读写可执行，128KB）。

**访问内存属性**：
- `ORIGIN(name)`：获取内存区域的起始地址。
- `LENGTH(name)`：获取内存区域的长度。
  例如：`_stack_top = ORIGIN(RAM) + LENGTH(RAM);` 将栈顶设置为 RAM 的末尾地址。

##### SECTIONS 命令
`SECTIONS` 命令是链接脚本的核心，用于定义输入段如何映射到输出段，以及输出段的内存布局。

**语法**：
```ld
SECTIONS
{
  . = address;  /* 设置位置计数器 */
  secname [address] [(type)] : [AT(lma)] [ALIGN(align)] [SUBALIGN(subalign)]
  {
    output-section-command
    output-section-command
    ...
  } [>region] [AT>lma_region] [=fillexp]
}
```

- **secname**：输出段的名称（如 `.text`、`.data`、`.bss`），必须符合输出文件格式（如 ELF）的约束。
- **address**：输出段的虚拟内存地址（VMA），可选。如果未指定，链接器根据位置计数器 `.` 或 `>region` 设置 VMA。
- **type**：段类型，可选：
  - `NOLOAD`：段不加载到内存（运行时不占用内存）。
  - `DSECT`、`COPY`、`INFO`、`OVERLAY`：向后兼容的类型，较少使用，标记段不可分配。
- **AT(lma)**：指定段的加载内存地址（LMA），用于 ROM 镜像场景（程序存储在 Flash，运行时复制到 RAM）。
- **ALIGN(align)**：强制输出段对齐到指定字节边界（如 `ALIGN(4)` 表示 4 字节对齐）。
- **SUBALIGN(subalign)**：强制输入段对齐，覆盖输入段的默认对齐。
- **>region**：指定段分配到的内存区域（由 `MEMORY` 定义）。
- **AT>lma_region**：指定加载地址的内存区域。
- **=fillexp**：指定填充值，用于填充段中的间隙（如 `=0x90909090`）。

**输出段命令（output-section-command）**：
- 符号赋值：如 `_start = .;`
- 输入段描述：指定哪些输入段合并到输出段。
- 直接数据：如 `LONG(0x1234)` 插入 4 字节数据。
- 特殊关键字：如 `KEEP`、`PROVIDE` 等。

**示例**：
```ld
SECTIONS
{
  . = 0x08000000;  /* 设置位置计数器 */
  .text :          /* 输出段 .text */
  {
    KEEP(*(.isr_vector))  /* 保留中断向量表 */
    *(.text*)            /* 所有代码段 */
    *(.rodata*)          /* 所有只读数据 */
    _etext = .;          /* 标记 .text 段结束地址 */
  } > FLASH

  .data : AT(_etext)  /* VMA 在 RAM，LMA 在 _etext */
  {
    _sdata = .;        /* 数据段起始 */
    *(.data*)          /* 所有初始化数据 */
    _edata = .;        /* 数据段结束 */
  } > RAM

  .bss (NOLOAD) :
  {
    _sbss = .;
    *(.bss*) *(COMMON) /* 未初始化数据和通用符号 */
    _ebss = .;
  } > RAM
}
```
**解释**：
- `.text` 段存储在 `FLASH`，包含中断向量表、代码和只读数据。
- `.data` 段运行时在 `RAM`，但初始值存储在 `FLASH`（从 `_etext` 开始）。
- `.bss` 段仅在 `RAM`，标记为 `NOLOAD`，不占用 `FLASH` 空间。
##### ENTRY 命令
`ENTRY(symbol)` 指定程序的入口点，即程序执行的第一条指令的地址。

**语法**：
```ld
ENTRY(symbol)
```

**优先级**（从高到低）：
1. `ld` 命令行 `-e` 选项。
2. 链接脚本中的 `ENTRY(symbol)`。
3. 如果定义了 `_start` 符号，使用其地址。
4. 如果存在 `.text` 段，使用其首地址。
5. 默认使用地址 `0x00000000`。

**示例**：
```ld
ENTRY(Reset_Handler)
```
将程序入口设置为 `Reset_Handler`（常见于嵌入式系统的复位中断处理函数）。

##### 输入段描述
输入段描述指定哪些输入文件的段合并到输出段，支持通配符和文件过滤。

**语法**：
```ld
FILENAME([EXCLUDE_FILE(FILENAME1 FILENAME2 ...) SECTION1 SECTION2 ...)
```

- **FILENAME**：输入文件名或通配符（如 `*` 表示所有文件，`main.o` 表示特定文件）。
- **EXCLUDE_FILE**：排除指定文件。
- **SECTION**：输入段名（如 `.text`、`.data`）或通配符（如 `.text*`）。

**示例**：
```ld
.text :
{
  KEEP(*(.isr_vector))        /* 保留所有文件的中断向量表 */
  *(.text*)                   /* 所有代码段 */
  main.o(.data)               /* main.o 的数据段 */
  *(EXCLUDE_FILE(crtend.o) .ctors) /* 排除 crtend.o 的 .ctors 段 */
}
```

**顺序规则**：
- `* (.text .data)`：按文件顺序，先处理每个文件的 `.text`，再处理 `.data`。
- `* (.text) * (.data)`：先处理所有文件的 `.text`，再处理所有文件的 `.data`。

**通用符号（COMMON）**：
通用符号（如未初始化的全局变量）通常放在 `.bss` 段：
```ld
.bss :
{
  *(.bss*) *(COMMON)
}
```

##### 符号赋值
链接脚本可以为符号分配地址，符号通常表示变量或函数的地址。

**语法**：
```ld
SYMBOL = EXPRESSION;
```
支持 C 风格运算符：`=`, `+=`, `-=`, `*=`, `/=`, `<<=`, `>>=`, `&=`, `|=`.

**特殊符号**：
- `.`：位置计数器，表示当前输出段的地址。
- `_etext`、`_sdata` 等：用户定义的符号，用于标记段边界。

**示例**：
```ld
_sdata = .;  /* 记录 .data 段起始地址 */
* (.data*)
_edata = .;  /* 记录 .data 段结束地址 */
```

**PROVIDE 关键字**：
`PROVIDE(symbol = expression)` 定义一个符号，仅当符号被引用但未在目标文件中定义时生效。
```ld
PROVIDE(etext = .);  /* 如果程序引用 etext 但未定义，则设置为 .text 段末尾 */
```

##### 其他常用命令
以下是一些重要的辅助命令：

1. **INCLUDE filename**：
   包含另一个链接脚本，类似 C 的 `#include`。
   ```ld
   INCLUDE common.lds
   ```

2. **INPUT(files)**：
   指定输入文件，文件会在 `-L` 指定的路径中搜索。
   ```ld
   INPUT(main.o startup.o)
   ```

3. **OUTPUT(filename)**：
   指定输出文件名，等同于 `ld -o`。
   ```ld
   OUTPUT(my_app.elf)
   ```

4. **OUTPUT_FORMAT(bfdname)**：
   设置输出文件格式（如 `elf32-littlearm`）。
   ```ld
   OUTPUT_FORMAT(elf32-littlearm)
   ```

5. **KEEP**：
   防止垃圾回收（`--gc-sections`）删除指定段。
   ```ld
   KEEP(*(.isr_vector))
   ```

6. **FILL(fillexp)**：
   设置段内未分配空间的填充值。
   ```ld
   .text : { *(.text) } =0x90909090
   ```

7. **ALIGN(exp)**：
   将位置计数器或表达式对齐到指定边界（必须为 2 的幂）。
   ```ld
   . = ALIGN(8);  /* 8 字节对齐 */
   ```

8. **OVERLAY**：
   定义多个段共享同一 VMA，运行时动态加载。
   ```ld
   OVERLAY 0x1000 : AT(0x4000)
   {
     .text0 { o1/*.o(.text) }
     .text1 { o2/*.o(.text) }
   }
   ```

##### VMA 和 LMA
每个输出段有两个地址：
- **VMA（Virtual Memory Address）**：运行时地址，即程序执行时的地址。
- **LMA（Load Memory Address）**：加载时地址，即程序存储时的地址。

**典型场景**：
- 程序存储在 Flash（LMA），运行时复制到 RAM（VMA）。
- 使用 `AT(lma)` 或 `AT>lma_region` 指定 LMA。
- 示例：
  ```ld
  .data : AT(_etext)
  {
    _sdata = .;
    *(.data*)
    _edata = .;
  } > RAM
  ```
  `.data` 段运行时在 RAM，初始值存储在 Flash（从 `_etext` 开始）。

**初始化代码**：
嵌入式系统中，启动代码需将 `.data` 从 LMA 复制到 VMA，并清零 `.bss`：
```c
void LowLevelInit(void) {
  extern uint32_t _etext, _sdata, _edata, _sbss, _ebss;
  uint32_t *src = &_etext, *dst = &_sdata;
  while (dst < &_edata) *dst++ = *src++;  // 复制 .data
  for (dst = &_sbss; dst < &_ebss; dst++) *dst = 0;  // 清零 .bss
}
```

#### 完整示例
以下是一个针对 STM32 微控制器的链接脚本示例：
```ld
MEMORY
{
  FLASH (rx) : ORIGIN = 0x08000000, LENGTH = 1024K
  RAM (rwx)  : ORIGIN = 0x20000000, LENGTH = 128K
}

ENTRY(Reset_Handler)

SECTIONS
{
  .text :
  {
    KEEP(*(.isr_vector))  /* 中断向量表 */
    *(.text*)            /* 代码 */
    *(.rodata*)          /* 只读数据 */
    _etext = .;          /* .text 段结束 */
  } > FLASH

  .data : AT(_etext)
  {
    _sdata = .;
    *(.data*)
    _edata = .;
  } > RAM

  .bss (NOLOAD) :
  {
    _sbss = .;
    *(.bss*) *(COMMON)
    _ebss = .;
  } > RAM

  .stack :
  {
    . = ALIGN(8);
    _sstack = .;
    . += 0x1000;  /* 分配 4KB 栈 */
    _estack = .;
  } > RAM

  .heap :
  {
    . = ALIGN(8);
    _sheap = .;
    . += 0x1000;  /* 分配 4KB 堆 */
    _eheap = .;
  } > RAM
}
```

**解释**：
- 定义 `FLASH` 和 `RAM` 内存区域。
- 设置入口点为 `Reset_Handler`。
- `.text` 段存储代码和只读数据，放在 `FLASH`。
- `.data` 段运行时在 `RAM`，初始值在 `FLASH`。
- `.bss` 段仅在 `RAM`，不占用 `FLASH`。
- 分配 4KB 的栈和堆空间。
#### 注意事项
- **对齐要求**：确保段和符号对齐（如 4 字节或 8 字节），避免运行时错误。
- **垃圾回收**：使用 `--gc-sections` 时，需用 `KEEP` 保留关键段（如中断向量表）。
- **符号冲突**：`PROVIDE` 定义的符号允许 C 代码重定义（无前导下划线）。
- **嵌入式系统**：常需手动复制 `.data` 和清零 `.bss`，确保启动代码正确。

# Makefile
这个工具就是Makefile，实际上能够完成这个任务的工具还有很多，Makefile只是在linux和unix系统上比较常见的方案。
我们实际执行的命令是make,它会根据相同目录下的makefile文件来构建项目，而这个makefile其实也是一个脚本文件，是由我们自行编写的。

在linux系统中，文件名是大小写敏感的，因此我们需要特定的makefile的文件名，具体为：
- GNUmakefile
- makefile
- Makefile
这三个，优先级由上到下。
为了方便讲解，我们同一使用makefile

makefile的基本内容为如下结构：
```makefile
target : dependencies
	command
```
具体来说，target是自己起的名字（符号）或者最终结果的文件名,dependencies是为了构建这个target所需要使用的其他的target或者文件名，command为这个target构建所需要执行的命令，例如：
```makefile
main.o: main.cpp
    g++ -c main.cpp -o main.o
```
而上述这个结构，被称为"规则"(rule)
> 上文所说的这个“自己起的名字”正规的叫法应该被称为伪目标，它表示不会生成这个名字的文件的目标，先按下不表

make的基本工作原理就是根据 makefile 中的规则，依次检查目标文件的时间戳和依赖文件的时间戳。如果依赖文件的时间戳比目标文件新，或者目标文件不存在，`make` 就会执行对应的命令来更新目标文件。`make` 会递归检查依赖关系，直到所有目标都更新完成。
我们在使用的时候通过
```bash
make <目标名>
```
来执行构建过程，比如上方这个我们可以
```bash
make main.o
```
也就是说假如我们有如下makefile：
```makefile
main.o: main.cpp
    g++ -c main.cpp -o main.o
main : main.o
	g++ main.o -o main
```
我们先自己把这个main程序构建出来
如果我们在该目录下执行`make main`，makefile就会检测main.cpp的时间戳和main.o的时间戳以及main这个可执行文件的时间戳，如果发现main.cpp的时间戳比main.o的新，或者main.o的时间戳比main的新，那make就会执行下方的命令来更新它们，更新的过程是递归的，因此当构建最后最终产物的时候一定是按照最新的main.cpp构建的。
当然，如果我们不对main.cpp做修改，是没有这个更新的，如果我们进行了修改，那结果就是会更新。

而除了按照文件名来声明规则，我们也可以使用伪目标来声明规则，简单来说就是这个目标只是有这个名字，是不是真的有这个文件makefile是不管的，当然，如果你把伪目标作为依赖来使用，那同样也是不会更新伪目标的。它只是用来方便我们执行一系列的命令而设置的，常见的比如clean和all，而伪目标需要使用.PHONY来声明比如：
```makefile
.PHONY: all clean
main.o: main.cpp
    g++ -c main.cpp -o main.o
main : main.o
	g++ main.o -o main
all : main
	touch main # 只是一个占位符
clean :
	rm -rf main.o main
```
这样我们执行`make all`和`make clean`就会分别产生对应的效果
> 如果你不指明目标名，而是直接执行make，make会读取makefile中的第一条规则来生成其中的第一个指定的文件。因此一般来说我们也会把all目标放在第一条规则。

makefile中的command就是一系列的命令。你写成这样
```makefile
.PHONY : helloworld
helloworld : 
	echo "helloworld"
```
你执行`make helloworld`同样也是可以正常执行echo的。
你会发现makefile会把自己执行的那个命令也一起输出出来，如果你不想它输出，就在前方加一个@
```makefile
.PHONY : helloworld
helloworld : 
	@echo "helloworld"
```


makefile同样支持变量，基本语法如下：
```makefile
变量名 = 值 # 定义
$(变量名) # 代换
```
而考虑到makefile的使用场景，这里的变量其实都是也都是用于规则中的，比如你可以这样：
```makefile
CC = g++
CFLAGS = -Wall -g
 
main.o: main.cpp
    $(CC) $(CFLAGS) -c main.cpp -o main.o
```
又或者你也可以这样
```makefile
obj = main.o
src = main.cpp
target = main
$(target) : $(obj)
	g++ $(obj) -o $(target)
$(obj) : $(src)
	g++ -c $(src) -o $(obj)
```
同时makefile也提供了很多默认变量，
在此列举几个常用的。

| 变量   | 含义（在配方里展开时的值） |
| ---- | ------------- |
| `$@` | 当前规则的目标文件名    |
| `$<` | 第一个依赖文件名      |
| `$^` | 全部依赖文件名（去重）   |
| `$+` | 全部依赖文件名（保留重复） |
| `$?` | 比目标新的依赖文件名列表  |

常见的用法比如：
```makefile
main.o: main.cpp
    g++ -c $^ -o $@
```

makefile同样也支持通配符规则，比如你可以这样来编译一个文件夹下的所有源文件
```makefile
%.o : %.cpp
	g++ -c $< -o $@
```

其实还有很多其他的变量，你可以通过`make -p`来查看全部的。

makefile同样支持条件运算和循环运算，不过对于我们来说就没有这一块的需求了，感兴趣的话可以自己去了解一下，包括前边所说的内置变量，其实是有特别多的，但是不知道也不影响使用。

需要说明的是，在目标中指定的文件名，是可以带有路径的，所以准确的来说其实是这个文件的路径名，这也就让我们可以分别去处理不同文件夹里的源文件，不过一般这种情况还是建议在子文件夹里多写一个makefile，能省很多事儿。
